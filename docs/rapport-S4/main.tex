\section{Introduction}
% ============================================================

Apr\`{e}s la mise en place de l'infrastructure (S2) et du microservice V\'{e}hicules (S3), la semaine~4 compl\`{e}te la plateforme m\'{e}tier. Deux nouveaux services ont \'{e}t\'{e} d\'{e}velopp\'{e}s --- \textbf{Conducteurs} et \textbf{Maintenance} --- et connect\'{e}s aux autres via Kafka. Une passerelle GraphQL (Apollo Server) unifie l'acc\`{e}s aux trois services, prot\'{e}g\'{e}e par Keycloak. Chaque service produit des donn\'{e}es de d\'{e}mo au d\'{e}marrage gr\^{a}ce \`{a} Datafaker, et la collection de tests Bruno couvre l'ensemble des endpoints.

\section{Nouveaux services m\'{e}tier}
% ============================================================

Les services \textbf{Driver} et \textbf{Maintenance} suivent la m\^{e}me structure que le service V\'{e}hicules : API REST Spring Boot, base PostgreSQL d\'{e}di\'{e}e, DDL g\'{e}r\'{e} par Hibernate (\texttt{ddl-auto: update}), et publication d'\'{e}v\'{e}nements Kafka. Une instance PostgreSQL unique h\'{e}berge les trois bases (\texttt{vehicle\_db}, \texttt{driver\_db}, \texttt{maintenance\_db}) avec isolation logique au niveau base de donn\'{e}es.

\subsection{Service Driver}

Le service g\`{e}re le cycle de vie des conducteurs : profils, permis, et coordination avec le service V\'{e}hicules lors des assignations. Le mod\`{e}le repose sur une relation \textbf{1-N} entre \texttt{Driver} et \texttt{DriverLicense}, avec suppression logique via \texttt{@SQLDelete} / \texttt{@SQLRestriction} Hibernate.

\begin{lstlisting}[language=Java, caption=Entit\'{e} Driver (extraits)]
@Entity @Table(name = "drivers")
@SQLDelete(sql = "UPDATE drivers SET deleted_at = NOW() WHERE id = ?")
@SQLRestriction("deleted_at IS NULL")
public class Driver {
    @Id @GeneratedValue(strategy = GenerationType.UUID)
    private UUID id;
    private String firstName, lastName, email, phone, employeeId;

    @Enumerated(EnumType.STRING)
    private DriverStatus status = DriverStatus.ACTIVE;

    @OneToMany(mappedBy = "driver", cascade = CascadeType.ALL)
    private List<DriverLicense> licenses = new ArrayList<>();
}
\end{lstlisting}

Une t\^{a}che \texttt{@Scheduled} (cron quotidien) d\'{e}tecte les permis expirant sous 30 jours et publie des \'{e}v\'{e}nements Kafka (\texttt{LICENSE\_EXPIRING}, \texttt{LICENSE\_EXPIRED}) consommables par les autres services.

\subsection{Service Maintenance}

Le service assure la tra\c{c}abilit\'{e} compl\`{e}te des interventions, de la planification \`{a} la cl\^{o}ture. L'entit\'{e} \texttt{MaintenanceRecord} utilise trois \'{e}num\'{e}rations (\texttt{MaintenanceType}, \texttt{MaintenanceStatus}, \texttt{MaintenancePriority}). Le lien vers le v\'{e}hicule est un simple UUID --- pas de cl\'{e} \'{e}trang\`{e}re inter-bases, conform\'{e}ment au principe microservices. Une t\^{a}che \texttt{@Scheduled} quotidienne (01h00) passe les maintenances en retard au statut \texttt{OVERDUE} et publie une alerte Kafka.

\subsection{Donn\'{e}es de d\'{e}mo (Datafaker)}

Plut\^{o}t que des fichiers \texttt{data.sql} statiques, chaque service embarque un \texttt{DataSeeder} activ\'{e} par le profil Spring \texttt{dev}. Le \texttt{vehicle-service} ins\`{e}re d'abord un v\'{e}hicule d\'{e}mo \`{a} UUID fixe via un \texttt{INSERT} SQL natif (n\'{e}cessaire car un UUID assign\'{e} manuellement ne passe pas par le cycle JPA \texttt{@GeneratedValue}), puis compl\`{e}te avec neuf v\'{e}hicules Faker si la base n'est pas d\'{e}j\`{a} peupl\'{e}e.

\section{Communication \'{e}v\'{e}nementielle entre services}
% ============================================================

\begin{infobox}
\textbf{Principe :} Saga par \textbf{chor\'{e}graphie} --- chaque service r\'{e}agit aux \'{e}v\'{e}nements des autres et publie ses propres r\'{e}sultats. Pas d'orchestrateur central.
\end{infobox}

Les \'{e}v\'{e}nements circulent dans une enveloppe commune \texttt{KafkaEventEnvelope} qui embarque l'identifiant, le type, l'horodatage, le payload m\'{e}tier et les m\'{e}tadonn\'{e}es de la source :

\begin{lstlisting}[language=java, caption=Enveloppe \'{e}v\'{e}nement unifi\'{e}e]
{
  "eventId": "625c87b9-...",
  "eventType": "MAINTENANCE_COMPLETED",
  "timestamp": 1775220800.293,
  "payload": { "vehicle_id": "39e75ab3-...", "cost_eur": 250.50 },
  "metadata": { "triggered_by": "maintenance-service" }
}
\end{lstlisting}

Le flux principal illustre la coordination entre les services via une saga compensatoire : Maintenance publie \texttt{MAINTENANCE\_STARTED} (T1), Vehicle consomme et passe en \texttt{IN\_MAINTENANCE} (T2). En cas d'\'{e}chec, Vehicle publie \texttt{MAINTENANCE\_REJECTED} qui d\'{e}clenche l'annulation de la fiche c\^{o}t\'{e} Maintenance (C1).

\section{Gateway GraphQL et authentification}
% ============================================================

\begin{infobox}
\textbf{Objectif :} Exposer une API GraphQL unifi\'{e}e (Apollo Server, Node.js, port 4000) devant les trois services m\'{e}tier, prot\'{e}g\'{e}e par des JWT Keycloak.
\end{infobox}

Le Gateway route les requ\^{e}tes via des clients REST Axios vers chaque service backend et \textbf{forwarde} le header \texttt{Authorization} re\c{c}u sans le modifier. Un intercepteur Axios assure la r\'{e}silience au d\'{e}marrage (\texttt{ECONNREFUSED} / \texttt{ECONNRESET}) avec retry automatique.

Les relations entre domaines sont r\'{e}solubles en une seule requ\^{e}te GraphQL : historique de maintenance depuis une fiche v\'{e}hicule, profil du conducteur assign\'{e}, technicien d'une intervention. Cela \'{e}vite au client d'interroger plusieurs services s\'{e}par\'{e}ment.

\subsection{Authentification Keycloak}

Keycloak (v26.5.7) est d\'{e}ploy\'{e} avec \texttt{--import-realm} pour charger automatiquement le realm \texttt{gestion-flotte} au d\'{e}marrage. Les services Spring Boot valident les JWT via \texttt{jwk-set-uri} ; le Gateway se contente de transmettre le token re\c{c}u.

\begin{lstlisting}[language=bash, caption=R\'{e}cup\'{e}ration du token (grant type password)]
POST /auth/realms/gestion-flotte/protocol/openid-connect/token
  grant_type=password  client_id=admin-cli
  username=test-user   password=password
\end{lstlisting}

La base de Keycloak diff\`{e}re selon le profil de d\'{e}ploiement : sous Kubernetes elle utilise \texttt{keycloak\_db} (cr\'{e}\'{e}e par le script \texttt{initdb} Helm) ; sous Docker Compose elle se connecte \`{a} la base \texttt{postgres} par d\'{e}faut.

\section{Qualit\'{e} : tests et couverture}
% ============================================================

\subsection{Collection Bruno}

La collection de tests remplace Postman par des fichiers \texttt{.bru} versionn\'{e}s dans Git. Un script \texttt{pre-request} dans \texttt{collection.bru} obtient et renouvelle le JWT automatiquement avant chaque requ\^{e}te via \texttt{bru.sendRequest}, en lisant la date d'expiration dans le payload du token --- aucune requ\^{e}te d'authentification manuelle n'est n\'{e}cessaire. Deux environnements sont fournis : \textbf{Docker} (\texttt{http://127.0.0.1:4000}, Keycloak port 8180) et \textbf{Minikube} (\texttt{http://flotte.local}, Ingress K8s).
